### 主题
Gemini Embedding 2 — Google 首个原生多模态 Embedding，5 种模态一个向量空间

### 事实清单
- 发布日期：2026-03-10
- 发布方：Google DeepMind
- 模型 ID：gemini-embedding-2-preview（公开预览）
- 定位：Google 首个原生多模态 Embedding 模型
- 支持模态：文本 / 图片 / 视频 / 音频 / PDF 文档（5种）
- 文本上限：8,192 tokens
- 图片上限：每次 6 张（PNG/JPEG）
- 视频上限：128 秒（MP4/MOV，H264/H265/AV1/VP9）
- 音频上限：80 秒（MP3/WAV）
- PDF 上限：6 页/请求
- 输出维度：默认 3072，支持 128/256/512/768/1536/2048（Matryoshka Representation Learning，无需重训练）
- 语言支持：100+ 语言
- 任务类型：8 种（SEMANTIC_SIMILARITY / CLASSIFICATION / CLUSTERING / RETRIEVAL_DOCUMENT / RETRIEVAL_QUERY / CODE_RETRIEVAL_QUERY / QUESTION_ANSWERING / FACT_VERIFICATION）
- 排行：MTEB 多语言 Top 5，多模态 SOTA（proprietary models）
- 定价：$0.20/百万 tokens（文本），Batch 半价
- 可用渠道：Gemini API + Vertex AI + LangChain / LlamaIndex / Haystack / Weaviate / Qdrant / ChromaDB
- 兼容性：与 gemini-embedding-001 不兼容，升级需重新 embed
- 来源：https://blog.google/innovation-and-ai/models-and-research/gemini-models/gemini-embedding-2/ / https://ai.google.dev/gemini-api/docs/embeddings

### 小红书 A 版（洞察向）

**标题**：Google 放大招：一个模型吃掉文字、图片、视频、音频、文档

Gemini Embedding 2 昨天发布，Google 第一个原生多模态 Embedding 模型。

**之前的世界**

搜索图片用图片模型，搜索文本用文本模型，搜索视频再换一个模型。每种模态一套向量空间，互不相通。想做"用文字搜视频"？得自己搭桥。

**现在的变化**

5 种模态（文字/图片/视频/音频/PDF）映射进同一个向量空间。意味着：
- 用一句话搜到相关视频片段
- 用一张图找到相关文档
- 用语音匹配相似的文本内容

不再需要为每种模态单独建索引。

**技术亮点**

Matryoshka Representation Learning——向量维度从 3072 可以缩到 128，不用重新训练。存储成本和检索速度可以按需调节，小场景用 128 维，高精度场景用 3072 维。

**为什么重要**

RAG 应用之前只能处理文本。现在一个 Embedding 模型就能把公司里的文档、录屏、会议录音、产品截图全部统一检索。企业知识库从"文本搜索"升级到"全模态搜索"。

MTEB 多语言排行 Top 5，定价 $0.20/百万 tokens，Batch 半价。性能和成本都到了实用区间。

通过 ofox.ai 接口调用 Gemini Embedding 2，可以直接在现有 RAG 流程中集成多模态检索能力。

### 小红书 B 版（专业向）

**标题**：Gemini Embedding 2 技术拆解：多模态统一向量空间实战指南

Google 3月10日发布 gemini-embedding-2-preview，首个原生多模态 Embedding 模型，公开预览。

**模型规格**

| 模态 | 上限 | 格式 |
|------|------|------|
| 文本 | 8,192 tokens | — |
| 图片 | 6 张/请求 | PNG/JPEG |
| 视频 | 128 秒 | MP4/MOV |
| 音频 | 80 秒 | MP3/WAV |
| PDF | 6 页/请求 | — |

输出维度默认 3072，通过 output_dimensionality 参数可调至 128/256/512/768/1536/2048。底层用 Matryoshka Representation Learning，截断低维不损失太多精度。

**8 种任务类型**

SEMANTIC_SIMILARITY / CLASSIFICATION / CLUSTERING / RETRIEVAL_DOCUMENT / RETRIEVAL_QUERY / CODE_RETRIEVAL_QUERY / QUESTION_ANSWERING / FACT_VERIFICATION

注意 CODE_RETRIEVAL_QUERY：代码检索有专属任务类型，说明 Google 在优化代码语义匹配。

**性能基准**

- MTEB 多语言排行 Top 5（文本）
- 多模态（图片/视频/音频）：proprietary models 中 SOTA
- 支持 100+ 语言语义理解

**定价**

文本 $0.20/百万 tokens，Batch 模式半价 $0.10/百万 tokens。图片/音频/视频按标准 Gemini 媒体 token 费率。

**生态集成**

已接入 LangChain / LlamaIndex / Haystack / Weaviate / Qdrant / ChromaDB / Vector Search。基本覆盖主流 RAG 技术栈。

**迁移注意**

gemini-embedding-001 和 gemini-embedding-2-preview 的向量空间不兼容。升级必须重新 embed 全部数据，不能混用。

**开发者视角**

多模态 Embedding 的实用门槛大幅降低。以前做"文搜图""图搜文"需要 CLIP + 自定义对齐层，现在一个接口搞定。对 RAG 应用来说，知识库从纯文本扩展到文档+截图+录屏+音频，检索质量取决于 Embedding 质量而不是工程复杂度。

通过 ofox.ai 统一接口调用 Gemini Embedding 2，可以快速在现有项目中集成多模态检索。

- 今日增长：+891 stars

### 小红书 A 版（洞察向）

**标题**：不装插件、不写代码，一句话让 AI 帮你操作网页

阿里开源了 PageAgent（alibaba/page-agent，3.9k⭐），一个"住在网页里"的 AI 助手。

**之前操控网页的方式**

想让 AI 自动填表单、点按钮、爬数据？
- 方案一：写 Selenium/Playwright 脚本 → 要会写代码
- 方案二：装 RPA 浏览器插件 → 依赖特定扩展
- 方案三：用 Headless Chrome → 要跑后端服务

三种方案都有门槛，而且都在网页"外面"隔靴搔痒。

**PageAgent 的思路**

直接把 Agent 注入到网页里面。一段 JS 脚本加载后，Agent 就像一个"看得懂页面的人"，你用自然语言说"帮我把这个表单填了"，它直接操作 DOM 完成。

不截图、不 OCR、不需要后端。你的登录态、Cookie 都直接复用，因为它就运行在你当前的浏览器页面里。

**三种使用方式**

1. 一行 CDN 脚本引入（最快体验）
2. npm 安装集成到自己的 SaaS 产品
3. 收藏夹书签栏一点即用（任何网页）

**为什么值得关注**

浏览器 Agent 赛道很热（Anthropic Computer Use、OpenAI Operator），但都是"从外面看屏幕"的思路。PageAgent 反过来，从网页内部操作 DOM，更轻量、更精准、延迟更低。

MIT 协议，BYOK 架构——模型用自己的 Key，数据不过第三方。通过 ofox.ai 接入 Qwen 或其他模型，就能零成本给现有网页加上 AI 操控能力。

### 小红书 B 版（专业向）

**标题**：PageAgent 技术拆解：阿里开源的浏览器内 GUI Agent 架构

alibaba/page-agent（3.9k⭐，MIT），纯客户端 JavaScript GUI Agent，自然语言控制网页界面。

**架构核心：从内部操作 DOM**

与主流浏览器 Agent（截图 → 视觉识别 → 模拟点击）不同，PageAgent 直接运行在网页内部：

```
用户自然语言指令
  → LLM 理解意图
  → 解析当前页面 DOM 结构（文本级，非截图）
  → 定位目标元素
  → 执行 DOM 操作（点击/填写/滚动）
```

优势：不依赖视觉模型，不需要截图和 OCR，精度高、延迟低。直接复用浏览器登录态和 Cookie。

**技术栈**

- 语言：TypeScript 82.1%
- 运行环境：浏览器端，纯客户端
- LLM 接入：BYOK 架构，支持 DashScope（Qwen）+ 任何 OpenAI 兼容接口
- 后端依赖：无（软件本身不含后端服务）
- 部署方式：CDN / npm / 书签栏 / Chrome 扩展

**四种集成方式**

| 方式 | 场景 | 复杂度 |
|------|------|--------|
| CDN 脚本 | 快速体验 | 低 |
| npm install | SaaS 产品集成 | 中 |
| 书签栏注入 | 任意网页使用 | 低 |
| Chrome 扩展 | 跨标签页操作 | 中 |

**vs 主流方案对比**

| 特性 | PageAgent | Anthropic Computer Use | OpenAI Operator |
|------|-----------|----------------------|-----------------|
| 运行位置 | 网页内部 | 外部 VM | 外部浏览器 |
| 感知方式 | DOM 文本 | 截图视觉 | 截图视觉 |
| 延迟 | 低 | 高 | 高 |
| 登录态 | 直接复用 | 需重新登录 | 需重新登录 |
| 后端需求 | 无 | 需要 | 需要 |
| 精度 | 高（直接定位） | 中（视觉识别） | 中（视觉识别） |

**开发者视角**

PageAgent 的轻量路线是个有趣的差异化。对 SaaS 开发者来说，`npm install page-agent` 然后给产品加一个 AI 副驾，比接入完整的浏览器自动化方案成本低很多。

限制也明显：纯 DOM 操作意味着对 Canvas/WebGL 渲染的页面无能为力，跨域 iframe 也有限制。适合表单密集型的管理后台和 SaaS 工具，不太适合游戏或复杂可视化页面。

通过 ofox.ai 接入 Qwen 或 GPT 系列模型，可以低成本为 PageAgent 提供 LLM 推理能力。
