## 260319-01 | GPT-5.4 mini & nano：子代理时代的平价引擎

### 小红书 A版（洞察向）

**标题**：7.6万张照片描述只要52美元？OpenAI的新模型改变了什么

用AI处理大量数据，最大的门槛不是技术，是成本。

3月17日，OpenAI发布了GPT-5.4 mini和nano两款小模型。不是旗舰的缩水版，而是专门为「子代理」场景设计的——让AI调用AI，批量干活。

💰 价格到底多便宜？

旗舰GPT-5.4：输入$2.50/百万token
mini：$0.75（便宜70%）
nano：$0.20（便宜92%）

Simon Willison实测：用nano给自己76000张照片写描述，总花费$52。平均每张不到0.07美分。

📊 便宜但不弱

mini在SWE-Bench Pro拿到54.4%（旗舰57.7%），OSWorld 72.1%（旗舰75.0%）。nano的SWE-Bench Pro也有52.4%，超过上一代mini。

速度？mini比上代快2倍。

🧠 为什么这件事重要？

以前用AI，一个请求对应一个模型。现在的趋势是「代理套代理」——一个主Agent拆分任务，派出几十个子Agent并行处理。这时候每个子Agent的成本就是关键。

nano的定价让这种架构在经济上真正可行了。

## 260319-02 | Mistral Forge + Small 4：企业自己造AI的新范式

### 小红书 A版（洞察向）

**标题**：不用OpenAI的模型，自己从零训练一个？Mistral说可以

大部分企业用AI的方式是：调用别人的模型，最多微调一下。

3月17日，Mistral在NVIDIA GTC上发布了Forge——一个让企业用自己的数据，从零开始训练AI模型的工具。

🔧 Forge做了什么？

不是微调，是全流程训练：
→ 预训练（用你的数据从头学）
→ 后训练（SFT/DPO对齐）
→ 强化学习（按你的规则优化）

配套还有合成数据管线和驻场工程师支持。

ASML（光刻机巨头）、Ericsson、欧洲航天局已经是早期客户。

🤖 同时发布的Small 4

119B参数，128个专家的MoE架构，每次只激活4个（6B参数）。

比上代延迟降40%，吞吐量提升3倍。Apache 2.0开源，定价$0.15/M tokens起。支持文本+图像，256K上下文。

🧠 为什么这条路线不一样？

OpenAI和Anthropic的逻辑是「用我们的模型」。Mistral的逻辑是「我帮你造你自己的模型」。

对于数据敏感的行业（芯片、航天、电信），把核心数据喂给第三方模型是不可接受的。Forge解决的就是这个问题。

CEO Arthur Mensch表示今年ARR将突破10亿美元。

## 260319-03 | Claude Dispatch：手机遥控桌面AI代理

### 小红书 A版（洞察向）

**标题**：出门在外，手机给电脑上的AI发任务？Anthropic做到了

你有没有过这种场景：人在外面，突然想让电脑上的AI帮你整理文件、查邮件、翻Notion笔记？

3月17日，Anthropic发布了Claude Dispatch（研究预览）。简单说：用手机远程操控桌面上的Claude Cowork。

📱 怎么用？

1. 更新桌面和手机端Claude应用
2. 手机扫QR码配对桌面
3. 在手机上打字发任务，Claude在桌面执行

整个过程是一个持续对话线程，上下文不丢失。你上午让它找文件，下午追问细节，它还记得。

🔒 安全怎么办？

破坏性操作（删文件、移目录）前，Claude会暂停并推送通知到手机，等你批准才继续。

⚠️ 但目前还不完美

MacStories实测14个任务：截图搜索、Notion摘要、邮件汇总都能做。但打开应用、发iMessage、跑Terminal命令全失败了。

而且速度偏慢——多个任务会排队串行处理。

🧠 为什么值得关注？

这是AI从「你坐在电脑前对话」到「你不在电脑前也能指挥它干活」的跨越。虽然是研究预览，但方向很明确：AI代理的工作不该被你的物理位置限制。

当前Max用户可用，Pro用户几天内覆盖。要求桌面保持唤醒且应用打开。

### 小红书 B版（专业向）

**标题**：Claude Dispatch技术解析：跨设备Agent的第一步

Anthropic在3月17日发布Dispatch（研究预览），将Claude Cowork从「桌面工具」升级为「跨设备Agent」。

**架构设计**

| 维度 | 细节 |
|------|------|
| 通信模型 | 手机↔桌面持久线程，上下文跨会话保持 |
| 配对方式 | QR码扫描绑定 |
| 执行环境 | 桌面端沙箱，可访问本地文件/连接器/插件 |
| 安全机制 | 破坏性操作前推送审批通知（Human-in-the-Loop） |
| 响应模式 | 异步——发送结果而非展示过程 |

**支持情况**

- 桌面：macOS / Windows x64（需保持唤醒+应用打开）
- 移动：iOS / Android
- 订阅：Max 即刻可用，Pro 几天内覆盖

**MacStories 14项实测**

| 结果 | 任务类型 |
|------|----------|
| ✅ 成功 | 截图文字搜索、Notion数据库摘要、邮件汇总、添加URL到Notion |
| ⚠️ 部分成功 | 按应用名找截图（匹配文件名而非视觉内容） |
| ❌ 失败 | 打开应用、iMessage发送、Terminal、AppleScript、Safari标签页 |

**当前限制**

- 单线程串行处理，速度较慢
- 无完成通知推送
- 不支持主动触发（只能被动响应）
- 不能访问Terminal和系统级操作

**开发者视角**

Dispatch解决的核心问题是Agent的「物理绑定」——你必须坐在执行环境前面。这是Agent从工具走向助理的关键一步。但当前的能力边界（不能操作系统级功能）说明沙箱安全和能力开放之间的平衡还在摸索。

对比OpenClaw的本地全权限模式，Anthropic选择了更保守的路线：先做安全，再开放能力。

## 260319-04 | Google官方出品：5种Agent Skill设计模式

### 小红书 B版（专业向）

**标题**：Google官方出品：5种Agent Skill设计模式深度拆解

**来源**：Google Cloud Tech 推荐 · Shubham Saboo (@Saboo_Shubham_) · X/Twitter · 阅读1百万+

**概述**：
SKILL.md 已被 Claude Code、Codex、ADK 等 30+ 工具采用，三层规范（Frontmatter→Body→Resources）不再是障碍。真正决定 Agent 行为质量的，是你往 Body 和 Resources 里装什么逻辑架构。Google Cloud Tech 推荐的这套框架，提炼出 5 种设计模式。

**5种模式一览**：
1. Tool Wrapper — 知识按需加载，不污染系统提示词
2. Generator — 模板+风格指南，结构质量双保证
3. Reviewer — 检查清单外置，换清单就换审查类型
4. Inversion — 先采访你再动手，三阶段门控
5. Pipeline — 硬性检查点，多步任务不许跳步

**模式① Tool Wrapper · 工具包装器**：
- 痛点：API文档太长塞不进prompt，硬编码浪费token
- 核心思想：专家知识存references/目录，用时才加载
- 运行机制：Agent写代码时触发→读取conventions→应用规则
- 关键细节：ADK用SkillToolset类实现动态加载
- 价值：token减半，Skill逻辑和领域知识彻底解耦

**模式② Generator · 生成器**：
- 痛点：同一任务每次输出格式不同，质量参差不齐
- assets/目录：放输出模板——控制结构和格式
- references/目录：放风格指南——控制语气和质量底线
- 执行流：读模板→读规范→问用户补变量→填充生成
- 价值：不只填模板，风格指南确保每次高质量输出

**模式③ Reviewer · 审查器**：
- 痛点：审查标准散落prompt里，换场景就要重写skill
- 分离策略：「查什么」写进checklist，「怎么查」写进skill
- Agent行为：读取checklist→逐条打分→按严重度分组
- 核心价值：换一份checklist就从代码审查变安全审计
- 复用潜力：一个Reviewer Skill，适配无限审查场景

**模式④ Inversion · 反转控制**：
- 痛点：Agent信息不全就动手，输出质量全凭运气
- 核心思想：翻转控制权——让Agent先当采访者
- 三阶段门控：Discovery（问目标和范围）→ Constraints（问限制和偏好）→ Synthesis（汇总后才执行）
- 关键指令：DO NOT start building until all phases complete
- 价值：最被低估的模式，却解决了Agent最大痛点——盲目行动

**模式⑤ Pipeline · 流水线**：
- 痛点：复杂任务Agent跳步，中间结果无法校验
- Stage 1 解析：拆解任务为子步骤，明确每步输入输出
- Gate检查点：用户确认后才放行，防止错误累积扩散
- Stage 2 生成：逐步执行，Gate可触发子Skill嵌套
- 终检组装：最强大但最重，适合多步骤高风险任务

**怎么选 · 需求→模式映射**：
- 产出型需求 → Generator为主，加Inversion收集变量
- 评估型需求 → Reviewer为主，嵌入Pipeline做门禁
- 复杂流程 → Pipeline为主，Gate嵌套独立子模式
- 知识密集 → Tool Wrapper为主，references/按需加载

**深度观点**：
模式本身不复杂，难在判断什么场景用什么组合。
- 通用语言：5种模式让团队有共同词汇讨论Skill设计
- 混合使用：实际项目中Pipeline+Reviewer+Generator经常组合
- 最被低估：Inversion看似简单，却解决了Agent最大痛点

**落地建议**：
1. 从Tool Wrapper开始 — 最简单，立刻减少prompt膨胀
2. 给Generator加风格指南 — 模板只管结构，风格指南管质量
3. 用 ofox.ai 实践这5种模式 — 直连Claude/GPT/Gemini，快速验证Skill设计
