## 260315-01

### 主题：Anthropic 100万token上下文标准化，长文本成本直降50%

### 事实清单
- Claude Opus 4.6 + Sonnet 4.6 的 1M token 上下文窗口以标准价格开放
- Opus 4.6：输入 $5/MTok，输出 $25/MTok；Sonnet 4.6：输入 $3/MTok，输出 $15/MTok
- 此前 Sonnet 4/4.5 超过 200K token 时输入价格翻倍、输出涨 50%
- MRCR v2 基准测试 Opus 4.6 得分 78.3%（前沿模型最高）
- PDF/图片上传从 100 页提升到 600 页
- 不再需要 beta header 即可发送超 200K 请求
- 来源：claude.com/blog/1m-context-ga，2026-03-13

### 小红书 A版（洞察向）

**标题**：100万token不加价，Anthropic在赌什么？

昨天 Anthropic 宣布：Claude Opus 4.6 和 Sonnet 4.6 的 100 万 token 上下文窗口，以标准价格开放。

之前怎样？用 Sonnet 4.5 处理超过 20 万 token 的长文本，输入价格翻倍，输出涨 50%。现在这笔溢价直接取消了。

**这意味着什么？**

一个完整的代码仓库、一本 600 页的 PDF、一整个季度的会议记录——都可以一次性塞进去，成本和处理短文本一样。

Opus 4.6 在长上下文基准测试 MRCR v2 上拿到 78.3%，是目前所有前沿模型的最高分。不只是窗口大，而是真的能在 100 万 token 里精准定位信息。

**为什么要这么做？**

长上下文是 Agent 的刚需。一个能浏览整个代码库的编程 Agent、一个能读完所有合同的法务 Agent——都需要大窗口。取消溢价，本质上是在降低 Agent 开发的门槛。

当竞争对手还在为超长上下文收溢价时，Anthropic 选择把它变成标配。

**对开发者的意义**：如果你在做 RAG，现在可以重新评估方案了——直接塞原文可能比切片检索更省事。

**标题**：Claude 1M Context GA：定价、性能与技术细节

Anthropic 正式将 100 万 token 上下文从 beta 转为 GA，覆盖 Opus 4.6 和 Sonnet 4.6。

| 模型 | 输入 | 输出 | 长上下文溢价 |

| Sonnet 4.5（旧）| $3→$6/MTok | $15→$22.5/MTok | >200K 时 2x/1.5x |

等效降幅：长文本场景输入成本降 50%，输出降 33%。

**性能基准**

- MRCR v2（长上下文检索）：Opus 4.6 达 78.3%，前沿模型最高

1. **RAG 架构决策**：对于 <1M token 的语料，直接全文输入可能优于分块检索
2. **Agent 开发**：全代码库理解、长对话记忆保持的成本大幅下降
3. **企业场景**：合同审查、财报分析等长文档处理变得经济可行

**注意**：Sonnet 4/4.5 仍需 beta header 且保留溢价。迁移到 4.6 版本是享受统一价的前提。

### 主题：Stanford研究：22-25岁开发者就业下降20%，AI对初级岗位冲击已成现实

### 事实清单
- 论文："Canaries in the Coal Mine? Six Facts about the Recent Employment Effects of AI"
- 作者：Erik Brynjolfsson, Bharat Chandar, Ruyu Chen（Stanford Digital Economy Lab）
- 论文发表于2025年8月，2026-03-06 SIEPR峰会更新展示
- 22-25岁软件开发者就业从2022年秋峰值下降约20%
- AI高暴露职业年轻工人整体相对下降13%
- 26岁以上开发者就业保持稳定或增长
- 数据源：ADP薪资微观数据（2500万+工人）
- AI暴露度量：Anthropic Economic Index
- Brynjolfsson原话："if this trend spreads to more occupations, we're going to see much more striking effects"

### 小红书 A版（洞察向）

**标题**：初级开发者就业暴跌20%，Stanford数据说了什么？

"AI 会不会抢走我的工作？"这个问题终于有了硬数据。

Stanford 数字经济实验室的研究团队，用美国最大薪资公司 ADP 的 2500 万工人数据做了追踪。结论很直接：

**22-25 岁软件开发者的就业量，从 2022 年秋季峰值下降了约 20%。**

同一时间，26 岁以上的开发者？就业稳定甚至增长。

**这不是"AI 替代开发者"，而是"AI 替代初级开发者做的事"。**

写样板代码、处理简单 bug、执行标准化任务——这些过去由初级工程师承担的工作，现在 AI 编程助手可以完成。企业不是不招人了，而是对入门级岗位的需求确实在减少。

更广泛的数据：所有 AI 高暴露职业中，年轻工人就业相对下降 13%。客服岗位也呈现类似模式。

研究者 Brynjolfsson 的判断："如果这个趋势扩散到更多职业，我们将看到更显著的影响。"

**对个人的意义**：初级岗位的竞争不再只是人和人之间的。能力定位要往 AI 做不好的方向靠——系统设计、跨团队协作、模糊需求的判断力。

### 小红书 B版（专业向）

**标题**：Stanford 就业研究拆解：谁在被 AI 替代？数据怎么说？

**论文**：Canaries in the Coal Mine（Stanford Digital Economy Lab，2025.08）
**作者**：Erik Brynjolfsson / Bharat Chandar / Ruyu Chen
**数据源**：ADP 薪资微观数据（覆盖 2500 万+ 美国工人，个体级别追踪）
**AI 暴露度量**：Anthropic Economic Index（区分"自动化型"和"增强型"AI 使用）

**核心发现**

| 群体 | 就业变化（2022峰值 → 2025.07）|
|------|-------------------------------|
| 22-25岁软件开发者 | 下降 ~20% |
| AI高暴露职业·年轻工人 | 相对下降 13% |
| 26岁以上开发者 | 稳定或增长 |
| 客服岗位 | 类似下降模式 |

**方法论亮点**

- 排除科技公司后结论不变
- 排除远程工作岗位后结论不变
- 排除 COVID 因素后结论不变
- 控制公司层面因素后，AI 高暴露职业年轻工人相对下降 13%

**解读**

这项研究的价值在于：用硬数据区分了"AI 焦虑"和"AI 现实"。不是所有开发者都受影响，而是特定年龄段 + 特定任务类型的岗位在收缩。

AI 生产力工具（如 Copilot）使现有高级开发者产出提升，减少了对初级执行型岗位的需求。这是典型的"增强型替代"——不是取消岗位，而是提高了入行门槛。
