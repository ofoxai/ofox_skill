### 小红书 A版（洞察向）

**标题**: 50 万行的 OpenClaw 太臃肿？这个以色列开发者用 4000 行重写了它

OpenClaw 是目前最火的开源 AI Agent，24.7 万星。但有个问题一直被吐槽：**50 万行代码、70 多个依赖、所有 Agent 跑在同一个进程里，安全靠的是应用层的白名单和配对码。**

以色列开发者 Gavriel Cohen 觉得这不够。他用 Claude Code 花了一个多月，写了一个叫 NanoClaw 的替代方案。

核心思路很简单：**每个 Agent 跑在自己的 Linux 容器里。**

不是应用层说「你不能访问这个文件」，而是操作系统层面直接隔离——文件系统、网络、进程，物理级别分开。就算一个 Agent 被攻破，也碰不到其他 Agent 的数据。

几个数据对比：

| | OpenClaw | NanoClaw |
|---|---------|----------|
| 代码量 | ~50 万行 | ~4000 行 |
| 配置文件 | 53 个 | 极少 |
| 依赖 | 70+ | 精简 |
| 安全模型 | 应用层隔离 | OS 容器隔离 |
| 全源码 token 数 | — | ~35k（Claude Code 可一次性读完） |

功能上没缩水：WhatsApp、Telegram、Discord、Slack、Gmail 都支持，有定时任务，有记忆系统，有 Web 访问。

已经有实际案例：Cohen 兄弟的 Qwibit AI 公司用 NanoClaw 搭了一个叫「Andy」的 Agent，每天早 9 点自动 briefing 销售管线状态，给团队分配任务。

**标题**: NanoClaw 技术拆解：4000 行 TypeScript + 容器隔离，如何替代 50 万行 OpenClaw

NanoClaw（17.8k⭐ / MIT）是 OpenClaw 的轻量安全替代方案。基于 Anthropic Claude Agent SDK 构建，核心理念：**OS 级容器隔离替代应用层隔离。**

**架构对比**：

| 维度 | OpenClaw | NanoClaw |
|------|---------|----------|
| 代码量 | ~500k 行 | ~4000 行核心 |
| 全源码 tokens | 巨大 | ~35k（适合 AI 辅助开发） |
| 技术栈 | Node.js + 复杂依赖链 | TypeScript 98.8% + Node.js 20+ |

| 安全模型 | 应用层（allowlist / pairing code） | Linux 容器（filesystem + network + process 隔离） |
| 配置文件 | 53 个 | 极简 |
| 依赖 | 70+ | Docker / Apple Container |
| 许可证 | Apache 2.0 | MIT |

**关键设计选择**：
- **每个 Agent 一个容器**：容器内只有 agentic loop + Agent SDK，进程级隔离
- **Agent 记忆**：per-group CLAUDE.md 文件，上下文不跨 Agent
- **Agent Swarms**：支持多 Agent 协作
- **安装方式**：`git clone` → Claude Code CLI `/setup`（Claude Code 自动化配置）

**支持渠道**：WhatsApp / Telegram / Discord / Slack / Gmail

**生产案例**：Qwibit AI 的 "Andy" Agent — 自动化销售管线管理，每日 9:00 briefing + 任务分配

**开发者视角**：

NanoClaw 的设计哲学是「小到 AI 能完全理解」。35k tokens 的全量源码意味着 Claude Code 可以一次性 pull 整个 codebase 进上下文，理解全部逻辑后 one-shot 大部分 feature。这对 AI 辅助开发是巨大优势——你改 OpenClaw 要读几天代码，改 NanoClaw 可能只要 8 分钟。

安全层面，容器隔离解决了 OpenClaw 最被诟病的问题：所有 Agent 共享内存、一个被攻破全部暴露。NanoClaw 的方案更接近生产级基础设施的思路。
