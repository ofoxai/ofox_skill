### 标题
Claude Code 核心开发者亲揭：AI 越用越省的 8 个秘密

### 正文

用 Claude 写代码，有没有发现：同样的问题，第一次慢，后面越问越快？

这背后是 Prompt Caching 技术。Claude Code 的核心开发者 @trq212，首次公开了他们怎么围绕这套技术设计整个产品的。

**【事实】**
193 万次阅读，597 次转发，8 条内部踩坑经验：

1️⃣ **静态内容放前面，动态内容放后面**
布局顺序：系统提示词 → 工具定义 → 项目文档 → 对话记录
越靠前的内容，被越多请求共享缓存，成本越低。

2️⃣ **用"插入消息"代替"修改提示词"**
时间变了、状态更新了？在下一条消息里加 `<system-reminder>` 标签，不要动系统提示词。改一个字，整个缓存就失效了。

3️⃣ **会话中途别换模型**
100k token 的对话从 Opus 换 Haiku，换完反而更贵——因为要给 Haiku 从头重建整个缓存。要换模型，用子 Agent 做交接，不要直接切。

4️⃣ **永远别中途增删工具**
工具定义是缓存前缀的一部分，加一个工具、删一个工具，整个缓存全部失效。

5️⃣ **Plan Mode 是工具调用，不是换工具集**
进入计划模式时，不要替换工具列表——调用 `EnterPlanMode` 这个工具本身。工具集始终不变，缓存始终有效。
额外收益：模型可以自主检测到复杂问题，自动进入 Plan Mode。

6️⃣ **工具太多？延迟加载，别删除**
MCP 工具几十个？发送轻量占位符，需要时再加载完整定义，缓存前缀永远稳定。

7️⃣ **上下文压缩时，参数必须和主对话一模一样**
上下文满了做压缩，如果换了 system prompt 或去掉工具定义，整个历史对话的缓存都白付了。正确做法：压缩请求和主对话用完全相同的前缀参数。

8️⃣ **监控缓存命中率，像监控宕机一样**
Claude Code 团队把缓存命中率下降当作 SEV（严重事件）处理，触发告警。

**【背后逻辑】**
这 8 条的核心是一个反直觉的原则：**整个系统架构，要围绕缓存前缀的稳定性来构建。**

大多数开发者的直觉是"按需给工具、按需换模型"，非常合理。
但在 Prompt Caching 体系里，这个直觉每次都在默默把钱烧掉。

**【对你的意义】**
如果你在用 Claude Code 或构建自己的 AI Agent：
- 检查你的提示词布局，静态内容是否在最前？
- 你有没有在会话中途换过模型或加过工具？
- 你是否监控了缓存命中率？

### 标题
Prompt Caching 工程实践：Claude Code 官方的 8 条踩坑经验

### 正文

整理自 @trq212（Claude Code 核心团队）首次公开的构建经验，适合在生产环境使用 Anthropic 接口的开发者。

**【核心原则】**
Prompt Caching 基于前缀匹配——从请求开始到每个 `cache_control` 断点，按相同前缀复用计算。前缀的任何变动使断点后的所有内容失效。

**【8 条工程经验】**

**1. 提示词布局：静态在前，动态在后**
Claude Code 的四层布局：
- Base System + Tools → globally cached
- CLAUDE.md → cached per project
- Session context → cached per session
- Conversation messages → grows each turn

**2. 用 Messages 传递更新，不要修改 System Prompt**
时间戳变更、状态更新 → 在下一轮 user message 或 tool result 中注入 `<system-reminder>` 标签，保持前缀不变。

**3. 不要在会话中途切换模型**
缓存是 model-specific 的。100k token 后从 Opus 切 Haiku，重建缓存的成本 > Opus 直接回答。
需要切换时：用 subagent，Opus 向子 Agent 传递 handoff message。

**4. 不要在会话中途增删工具**
工具定义在 cached prefix 中，任何变动使整个 prefix 后的内容失效。

**5. Plan Mode：状态转换用工具建模**
- 错误做法：进入 Plan Mode 时替换工具集（只保留只读工具）
- 正确做法：全程保持完整工具集，用 `EnterPlanMode` / `ExitPlanMode` 作为工具

附加收益：模型可以自主检测到复杂问题并调用 `EnterPlanMode`，无需用户触发，且无缓存中断。

**6. 大量 MCP 工具：defer_loading 而非删除**
每个工具发送轻量 stub（仅工具名 + `defer_loading: true`），模型通过 ToolSearch 按需加载完整 schema。前缀中永远有相同 stubs，顺序固定，缓存稳定。

**7. Compaction（上下文压缩）：cache-safe forking**
- 错误实现：新 system prompt + 无工具 → 全量重新付费
- 正确实现：与主对话完全相同的 system prompt + tools + 历史消息，末尾追加 compaction prompt

结果：只有 compaction prompt 本身是新 token。
注意：需预留 "compaction buffer"，保证 context window 有足够空间容纳 compact 消息和输出 token。

**8. 监控缓存命中率**
Claude Code 将缓存命中率纳入 SLO，命中率下降 = 触发 SEV 告警。
监控指标：`cache_creation_input_tokens` vs `cache_read_input_tokens` 比例。

**【开发者行动项】**
- 初始化阶段：确认提示词排序（静态→动态）
- 架构阶段：所有状态转换用消息/工具建模，不修改前缀
- 运营阶段：监控 cache read/creation 比例，设置下降告警

Claude Code's caching infrastructure, from the builder (@trq212):

Cache hit rate = uptime. They declare SEVs when it drops.

8 lessons learned:

1/ Static prompt first, dynamic last.
Layout: Base System → Tools → CLAUDE.md → Session → Messages

2/ Never edit system prompts to update state.
Use messages + <system-reminder> tags instead.

3/ Don't switch models mid-session.
100k tokens into Opus → switching to Haiku costs MORE due to cache rebuild.

4/ Never add/remove tools mid-session.
Tools are part of the cached prefix. Any change = full invalidation.

5/ Plan mode = tool call, not swapping the tool set.
EnterPlanMode is a tool the model can call itself — no cache break.

6/ Too many MCP tools? Defer-load stubs, never remove mid-session.

7/ Compaction must reuse parent's exact prefix.
Same system prompt + tools + history. Only the compaction prompt is new tokens.

8/ Monitor cache hit rate like uptime.
A few % of cache miss = dramatic cost and latency impact.

The mental model: design your entire system around prefix stability.
